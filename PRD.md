**版本：** v0.1  
**日期：** 2026-04-19  
**产品形态：** 微信小程序  
**阶段定位：** MVP 需求文档

---

## 1. 可行性结论

电子衣橱类微信小程序可行，适合先从“衣物管理 + 半智能搭配”切入。最关键的判断是：MVP 不建议一开始做高成本的虚拟试衣或复杂 3D 换装，而是先把衣物图片沉淀为结构化标签，再用规则 + 大模型生成搭配建议。

**推荐路线：**

1. 用户上传衣物图片，小程序完成压缩、存储、基础内容安全检测。
2. 通过图像识别/多模态模型提取衣物分类、颜色、季节、场景、风格标签。
3. 用户可手动修正标签，形成个人衣橱数据资产。
4. 搭配推荐基于衣物标签、天气/温度、场景、用户偏好生成 3-5 套方案。
5. 根据用户收藏、跳过、替换、实际穿着反馈，逐步优化推荐。

**可行原因：**

- 微信小程序适合轻量工具，用户无需下载 App，适合拍照上传和日常查看。
- 衣物管理的核心数据结构相对明确：品类、颜色、季节、场景、风格、图片。
- 智能搭配可以先做“标签规则 + LLM 文案解释”，不必依赖复杂模型训练。
- 图片存储、云函数、数据库、图像识别、内容安全、LLM 接口都有成熟云服务可接入。

**主要难点：**

- 用户首次建衣橱的上传成本高，需要批量上传、自动识别、快速编辑降低阻力。
- 图片识别准确率不稳定，尤其是叠穿、复杂花纹、光线差、非平铺照片。
- 搭配“好不好看”主观性强，需要用户反馈闭环，而不是只靠一次性 AI 输出。
- 用户照片和衣物图片涉及隐私，需要明确授权、删除、导出和安全策略。

## 2. 产品定位

### 2.1 产品名称

暂定：**智搭衣橱**

### 2.2 一句话描述

帮用户把自己的衣服快速整理成电子衣橱，并基于天气、场景和个人偏好生成可直接穿的搭配方案。

### 2.3 目标用户

| 用户类型 | 特征 | 核心诉求 |
| --- | --- | --- |
| 通勤女性/男性 | 衣物数量较多，每天需要搭配 | 快速决定今天穿什么 |
| 穿搭爱好者 | 关注风格、季节、场景搭配 | 管理衣橱、沉淀 lookbook |
| 轻断舍离用户 | 想知道哪些衣服闲置 | 统计衣物使用频次，辅助整理 |
| 出差/旅行用户 | 有行程打包需求 | 根据天数、天气、场景生成行李搭配 |

### 2.4 产品边界

**MVP 做：**

- 衣物上传与分类管理
- AI 辅助打标签
- 手动编辑衣物信息
- 按场景/天气生成搭配
- 搭配收藏与反馈

**MVP 不做：**

- 复杂虚拟试衣
- 真实人体换装生成
- 电商导购闭环
- 社区内容流
- 二手交易

## 3. 产品目标与指标

### 3.1 阶段目标

| 阶段 | 目标 | 判断标准 |
| --- | --- | --- |
| MVP | 验证用户愿不愿意上传衣物并使用搭配推荐 | 用户上传 20 件以上衣物，且每周至少使用 2 次推荐 |
| Beta | 验证搭配结果是否有用 | 推荐收藏率、替换率、满意度反馈稳定 |
| 商业化 | 验证付费点 | 订阅、AI 次数包、穿搭报告或高级功能转化 |

### 3.2 核心指标

| 指标 | 说明 | MVP 目标 |
| --- | --- | --- |
| 新用户建档完成率 | 完成基础偏好 + 上传至少 10 件衣物 | >= 35% |
| 人均衣物数 | 每个活跃用户沉淀的衣物数量 | >= 25 件 |
| 搭配生成使用率 | 活跃用户中使用搭配推荐的人占比 | >= 40% |
| 搭配收藏率 | 生成方案被收藏的比例 | >= 15% |
| 推荐差评率 | 用户标记“不合适”的比例 | <= 30% |
| 次月留存 | 工具类小程序的留存关键指标 | >= 15% |
| 单次搭配成本 | AI 与云资源成本 | MVP 尽量控制在 0.01 元以内，不含图片生成 |

## 4. 核心用户场景

### 4.1 整理衣橱

用户打开小程序，拍照上传上衣、裤子、裙子、鞋包配饰。系统自动识别品类、主色、季节和适合场景，用户快速确认或修改，形成电子衣橱。

### 4.2 每日搭配

用户选择“通勤”“约会”“运动”“正式”等场景，系统结合当前天气/温度和用户衣橱，推荐 3-5 套搭配，并说明为什么适合。

### 4.3 指定单品搭配

用户想穿某件外套或新买的裙子，选择该单品后，系统从衣橱中推荐可搭配的内搭、下装、鞋包和配饰。

### 4.4 旅行打包

用户输入目的地、天数、天气和场景，系统生成少件多搭的行李清单和每日搭配。

### 4.5 衣橱复盘

用户查看本月常穿、闲置、重复品类、颜色分布和缺口建议，辅助断舍离或理性购买。

## 6. 用户流程

### 6.1 新用户首次使用

1. 进入小程序。
2. 同意隐私政策和用户协议。
3. 选择风格偏好：简约、通勤、休闲、甜美、运动、街头、正式等。
4. 选择常用场景：上班、上学、约会、聚会、旅行、运动等。
5. 上传首批衣物，建议引导用户至少上传 10 件。
6. 系统自动识别标签，用户快速确认。
7. 首页展示衣橱概览和“生成今日搭配”入口。

### 6.2 衣物上传流程

1. 用户点击“添加衣物”。
2. 选择拍照或相册。
3. 前端压缩图片并生成缩略图。
4. 服务端保存原图/压缩图到对象存储。
5. 调用内容安全检测。
6. 调用图像识别/多模态模型生成标签。
7. 返回识别结果，用户确认或修改。
8. 入库衣物信息。

### 6.3 搭配推荐流程

1. 用户选择场景、温度、风格偏好。
2. 可选：指定一件必须出现的单品。
3. 系统筛选可用衣物：季节、场景、品类、颜色、状态。
4. 规则引擎先生成候选组合。
5. LLM 对候选组合进行排序、解释和补充。
6. 返回 3-5 套搭配。
7. 用户收藏、替换某件、重新生成或反馈“不合适”。

## 9. 智能搭配策略

### 9.1 MVP 推荐策略

MVP 使用三层策略：

1. **硬规则过滤**
   - 季节/温度不匹配的衣物剔除。
   - 场景不匹配的衣物降权或剔除。
   - 状态为洗涤中、淘汰、隐藏的衣物剔除。

2. **搭配规则生成候选**
   - 品类组合：上衣 + 下装 + 鞋；连衣裙 + 外套 + 鞋；外套 + 内搭 + 下装 + 鞋。
   - 颜色规则：同色系、邻近色、基础色平衡、主色不超过 3 个。
   - 风格规则：正式场景减少运动/夸张印花，休闲场景放宽。

3. **LLM 排序与解释**
   - 输入结构化衣物标签，不优先传原图，降低隐私和成本。
   - 输出搭配标题、适用场景、推荐理由、替换建议。

### 9.2 不建议 MVP 直接做的能力

| 能力 | 原因 |
| --- | --- |
| 真人虚拟试衣 | 生成成本高，效果不稳定，隐私风险更高 |
| 自动抠图合成全身 look | 需要较多视觉工程和模型调优 |
| 个性化训练模型 | 冷启动数据不足，前期投入产出不高 |
| 社区推荐流 | 内容审核、运营和增长成本会明显增加 |

## 10. 技术方案建议

### 10.1 架构

```text
微信小程序
  -> API 网关/云函数
  -> 数据库：用户、衣物、搭配、日志
  -> 对象存储：衣物原图、压缩图、缩略图
  -> AI 服务：图像识别、多模态理解、LLM 搭配推荐
  -> 内容安全：图片审核、文本审核
  -> 任务队列：异步识别、批量上传处理
```

### 10.2 推荐技术选型

| 模块 | 建议 |
| --- | --- |
| 前端 | 微信小程序原生开发或 Taro/uni-app |
| 后端 | 微信云开发 CloudBase，或 Node.js + 腾讯云/阿里云 |
| 数据库 | CloudBase 数据库/MongoDB；后期可补 MySQL |
| 图片存储 | COS/云存储，生成缩略图，避免原图频繁加载 |
| AI 标签 | 图像识别 API + 规则归一化 |
| 搭配生成 | 规则引擎 + LLM |
| 内容安全 | 微信/云厂商图片与文本安全接口 |
| 监控 | 云函数日志、AI 调用日志、成本看板 |

## 11. 成本拆解

### 11.1 一次性成本

| 成本项 | 说明 | 粗略范围 |
| --- | --- | --- |
| 产品设计 | PRD、原型、交互、视觉规范 | 0.5-3 万元，取决于是否外包 |
| 小程序前端 | 页面、组件、上传、筛选、搭配结果 | 2-8 万元 |
| 后端开发 | 用户、衣物、搭配、AI 调用、日志 | 3-10 万元 |
| AI 策略开发 | 标签体系、推荐规则、Prompt、评测集 | 2-8 万元 |
| 测试与上线 | 兼容性、性能、审核、埋点 | 0.5-3 万元 |
| 小程序认证 | 企业主体通常需要认证/年审，常见口径约 300 元/次，以微信公众平台后台为准 | 约 300 元/年 |
| 域名/证书 | 自建后端需要域名和 HTTPS；云开发可减少这部分 | 0-1000 元/年 |

**MVP 外包粗估：** 5-15 万元。  
**较完整版本：** 15-40 万元。  
**包含高质量虚拟试衣/图片生成：** 通常需要单独评估，可能显著高于基础版本。

### 11.2 持续运营成本

| 成本项 | 计费逻辑 | 影响因素 |
| --- | --- | --- |
| 图片存储 | 按容量计费 | 用户数、人均衣物数、图片压缩策略 |
| 图片流量/CDN | 按外网下行流量计费 | 用户浏览频次、缩略图尺寸、缓存策略 |
| 云函数/后端计算 | 按调用次数、计算资源或套餐计费 | 上传、筛选、搭配、统计调用量 |
| 数据库 | 容量、读写、备份 | 衣物数量、日志保留周期 |
| 图像识别 | 通常按图片调用次数计费 | 新增衣物数量、是否重复识别 |
| LLM 搭配 | 按 token 或调用量计费 | 搭配生成次数、输入衣物数量、输出长度 |
| 内容安全 | 按接口/次数/套餐，或平台能力限制 | 用户上传图片和文本数量 |
| 运维监控 | 日志、告警、备份 | 用户规模、稳定性要求 |
| 客服与运营 | 用户问题、活动、社群 | 增长阶段会明显增加 |
| 市场推广 | 投放、KOL、种子用户激励 | 增长目标 |

### 11.3 成本测算样例

假设：

- 1000 MAU。
- 人均上传 30 件衣物。
- 单张压缩图 300KB，缩略图 80KB。
- 每月新增上传 5000 张。
- 每月搭配生成 10000 次。
- MVP 只做标签识别和文字/结构化搭配，不做图片生成。

粗略估算：

| 项目 | 估算方式 | 量级判断 |
| --- | --- | --- |
| 图片存储 | 约 30-40GB 起步，加上备份和缩略图 | 通常几十元/月级别，取决于云厂商计费 |
| 图片识别 | 新增图片数 * 单次识别价格 | 5000 张通常为几十元/月级别 |
| LLM 搭配 | 调用次数 * 单次 token 成本 | 10000 次可能为几十元到数百元/月 |
| 云函数/数据库 | 套餐或按量 | 早期通常几十元到数百元/月 |
| 流量 | 缩略图访问量 * 图片大小 | DAU 和浏览深度上来后会成为主要成本 |

**结论：** 早期 MVP 的云和 AI 成本通常不是最大项，最大项是开发成本和冷启动运营成本。真正容易失控的是图片生成/虚拟试衣、原图大流量访问、无缓存的反复 AI 识别。

## 12. 商业化方向

| 模式 | 说明 | 适合阶段 |
| --- | --- | --- |
| 免费 + 次数限制 | 每月免费生成一定次数搭配 | MVP/Beta |
| 会员订阅 | 无限搭配、高级统计、旅行搭配、风格报告 | Beta 后 |
| AI 次数包 | 高级 AI、虚拟试衣、图片生成按次收费 | 有明确成本后 |
| 导购分佣 | 缺少单品推荐购买链接 | 后期，需要谨慎避免影响工具信任 |
| B 端合作 | 造型师、服装品牌、衣橱整理师工具 | 后期 |

MVP 不建议过早导购化。电子衣橱的核心信任来自“用我已有的衣服搭好看”，如果一开始过重卖货，用户可能不愿意认真录入衣橱。

## 13. 风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 用户不愿意上传太多衣服 | 首次建档流失 | 首批目标设为 10 件，支持批量上传和快速确认 |
| AI 标签不准 | 分类错误、颜色错误 | 标签可编辑，展示 AI 置信度，收集纠错数据 |
| 推荐结果不好看 | 用户不收藏、不复用 | 先用规则兜底，再让 LLM 解释和排序 |
| 图片成本上升 | 存储和流量变高 | 强制压缩、缩略图优先、原图冷存储 |
| 隐私顾虑 | 用户不愿上传照片 | 不要求真人照，默认只上传衣物；提供删除和导出 |
| 内容审核 | 用户上传违规图 | 上传后先过内容安全，异常图片隔离 |
| 微信审核 | 功能描述、隐私收集不合规 | 上线前准备隐私协议、数据用途说明、用户授权流程 |

## 14. 隐私与合规

MVP 必须明确：

- 用户上传图片的用途：仅用于衣物管理、识别和搭配推荐。
- 默认不要求上传真人照片。
- 支持用户删除衣物和删除账号数据。
- 对 AI 调用做最小化传输：搭配阶段尽量使用结构化标签，不传原图。
- 隐私政策中说明第三方云服务和 AI 服务的使用范围。
- 如接入定位天气，需单独申请定位授权，并提供手动城市选择替代方案。
- 如自建服务部署在中国大陆服务器，需要关注 ICP 备案、域名和数据合规要求。

## 16. 验收标准

### 16.1 衣物管理

- 用户可成功上传图片并创建衣物。
- 图片压缩后可正常展示缩略图。
- 衣物可按品类、颜色、季节筛选。
- 用户可编辑 AI 自动识别的标签。
- 用户可删除衣物，删除后列表和推荐不再使用该衣物。

### 16.2 AI 标签

- 支持至少 8 个一级品类：上衣、下装、连衣裙、外套、鞋、包、配饰、其他。
- 支持至少 12 个颜色标签。
- 支持季节和场景标签。
- AI 失败时不阻塞用户创建，可进入“待补充标签”状态。

### 16.3 智能搭配

- 用户可基于场景生成至少 3 套搭配。
- 用户可指定单品生成搭配。
- 推荐结果必须只使用用户衣橱中存在且状态可用的衣物。
- 推荐结果包含搭配说明和替换建议。
- 用户可收藏、跳过、反馈不合适。

### 16.4 性能与稳定性

- 衣橱列表首屏加载目标 < 2 秒。
- 单次搭配生成目标 < 8 秒。
- 图片上传失败有重试和明确提示。
- AI 接口失败时有兜底文案和重试入口。
